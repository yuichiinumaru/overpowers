## 🎯 核心原则

**所有任务必须遵循以下流程：**

```
1️⃣ 计划书 → 2️⃣ 用户确认 → 3️⃣ 执行 → 4️⃣ 汇报
```

**禁止跳过任何步骤！**

---

## 📋 标准工作流程

### 阶段 1: 计划书 (PM 阶段)

**触发条件：** 用户提出新任务

**执行步骤：**

1. **需求分析**
   - 理解任务目标
   - 识别关键需求
   - 评估优先级

2. **技术设计**
   - 选择技术方案
   - 设计执行步骤
   - 评估风险和备选方案

3. **时间计划**
   - 估算各阶段耗时
   - 设定里程碑
   - 明确交付物

4. **生成计划书**
   - 使用标准模板
   - 清晰直观（Markdown 表格 + 列表）
   - 包含确认提示

**输出格式：**

```markdown
## 📋 [任务名称] 计划书

### 🎯 任务目标

| 项目 | 内容 |
|------|------|
| **任务类型** | [类型] |
| **优先级** | P0/P1/P2 |
| **预计耗时** | X 分钟 |

### 📐 执行步骤

```
1️⃣ 步骤 1
       ↓
2️⃣ 步骤 2
       ↓
3️⃣ 步骤 3
```

### ⏱️ 时间计划

| 阶段 | 预计时间 |
|------|----------|
| **准备** | X 分钟 |
| **执行** | X 分钟 |
| **验证** | X 分钟 |

**总耗时：** X 分钟

### 📁 交付物

| 交付物 | 位置 |
|--------|------|
| **文件 1** | `路径/` |
| **文件 2** | `路径/` |

---

**X，请回复 "确认执行" 开始任务！** 🦞

**最强大脑 AI 公司 敬上**
```

---

### 阶段 2: 用户确认 (PM 阶段)

**触发条件：** 计划书已发送

**等待用户回复：**

| 回复 | 行动 |
|------|------|
| **"确认执行"** | 进入执行阶段 |
| **提出修改** | 修改计划书，重新确认 |
| **取消任务** | 终止任务，记录原因 |

**禁止行为：**
- ❌ 未确认就执行
- ❌ 催促用户确认
- ❌ 擅自修改已确认的计划

---

### 阶段 3: 执行 (BE/Ops 阶段)

**触发条件：** 用户确认执行

**执行步骤：**

1. **按计划书执行**
   - 严格遵循计划步骤
   - 如有变更，先汇报再调整

2. **实时记录**
   - 记录执行日志
   - 保存中间结果
   - 标记问题和解决方案

3. **异常处理**
   - 遇到问题立即汇报
   - 提供解决方案选项
   - 等待用户决策

**输出格式：**

```markdown
## 🎯 执行阶段

### 步骤 1️⃣: [步骤名称] (角色)

[执行命令/操作]

[执行结果]

### 步骤 2️⃣: [步骤名称] (角色)

...
```

---

### 阶段 4: 汇报 (PM/QA 阶段)

**触发条件：** 任务执行完成

**汇报内容：**

1. **执行结果**
   - 成功率统计
   - 交付物清单
   - Git 提交信息

2. **问题分析**
   - 遇到的问题
   - 解决方案
   - 经验教训

3. **后续建议**
   - 下一步计划
   - 优化建议
   - 风险提示

**输出格式：**

```markdown
## ✅ [任务名称] 完成！

### 📊 执行结果

| 指标 | 目标 | 实际 | 状态 |
|------|------|------|------|
| **指标 1** | X | X | ✅/❌ |
| **指标 2** | X | X | ✅/❌ |

### 📁 交付物

| 文件 | 位置 |
|------|------|
| **文件 1** | `路径/` |
| **文件 2** | `路径/` |

### 📂 Git 提交

| 项目 | 详情 |
|------|------|
| **提交 ID** | `xxx` |
| **提交信息** | xxx |
| **推送状态** | ✅ 成功 |

### 🔄 下一步

1. **建议 1**
2. **建议 2**

---

**X，任务已完成！** 🦞

**最强大脑 AI 公司 敬上**
```

## 🎭 角色分工

### 产品经理 (PM)

**职责：**
- 需求分析和计划书编写
- 用户沟通和确认
- 最终汇报和文档整理

**使用场景：**
- 任务开始（计划书）
- 任务结束（汇报）
- 异常处理（方案选择）

---

### 后端工程师 (BE)

**职责：**
- 技术方案实现
- 代码开发和脚本编写
- 数据处理和 API 调用

**使用场景：**
- 执行阶段（技术开发）
- 问题排查（日志分析）

---

### 运维工程师 (Ops)

**职责：**
- 环境配置和部署
- 系统监控和日志
- Git 提交和推送

**使用场景：**
- 执行阶段（环境准备）
- 执行阶段（服务部署）
- 汇报阶段（Git 操作）

---

### 测试工程师 (QA)

**职责：**
- 结果验证
- 质量检查
- 文档审核

**使用场景：**
- 执行阶段（中间验证）
- 汇报阶段（最终验证）

---

## 📝 文档规范

### 计划书模板

**必须包含：**

1. ✅ 任务目标（表格）
2. ✅ 执行步骤（流程图）
3. ✅ 时间计划（表格）
4. ✅ 交付物清单（表格）
5. ✅ 确认提示（加粗）

**禁止：**
- ❌ 冗长文字描述
- ❌ 模糊的时间估算
- ❌ 缺少确认提示

---

### 执行日志模板

**必须包含：**

1. ✅ 步骤编号（emoji）
2. ✅ 执行角色
3. ✅ 命令/操作
4. ✅ 执行结果
5. ✅ 错误信息（如有）

**格式：**

```markdown
### 步骤 1️⃣: [步骤名称] (角色)

[命令/操作]

[结果]
```

---

### 汇报模板

**必须包含：**

1. ✅ 执行结果统计（表格）
2. ✅ 交付物清单（表格）
3. ✅ Git 提交信息（表格）
4. ✅ 问题分析（如有）
5. ✅ 后续建议

**禁止：**
- ❌ 只报喜不报忧
- ❌ 缺少数据统计
- ❌ 没有后续建议

---

## ⚠️ 禁忌行为

### 严格禁止

| 行为 | 后果 | 替代方案 |
|------|------|----------|
| **未确认就执行** | 违反核心原则 | 等待用户确认 |
| **跳过计划书** | 用户不知情 | 必须先生成计划书 |
| **隐瞒问题** | 失去信任 | 立即汇报 + 解决方案 |
| **擅自修改计划** | 超出授权范围 | 先汇报，等确认 |
| **重复回复** | 用户体验差 | 使用 NO_REPLY 标记 |

## 🔄 异常处理流程

### 场景 1: 执行失败

**处理步骤：**

```
1. 立即停止执行
   ↓
2. 记录错误信息
   ↓
3. 分析失败原因
   ↓
4. 提供解决方案选项
   ↓
5. 等待用户决策
   ↓
6. 按决策执行
```

**汇报格式：**

```markdown
## ❌ [步骤名称] 执行失败

### 问题描述

[错误信息]

### 原因分析

| 原因 | 说明 |
|------|------|
| **原因 1** | xxx |
| **原因 2** | xxx |

### 解决方案

| 方案 | 说明 | 预计耗时 |
|------|------|----------|
| **方案 A** | xxx | X 分钟 |
| **方案 B** | xxx | X 分钟 |

**X，请选择方案 A/B/C！** 🦞
```

---

### 场景 2: 需求变更

**处理步骤：**

```
1. 记录变更内容
   ↓
2. 评估影响（时间/资源）
   ↓
3. 更新计划书
   ↓
4. 重新确认
   ↓
5. 按新计划执行
```

---

### 场景 3: 超时/延期

**处理步骤：**

```
1. 提前预警（完成 50% 时）
   ↓
2. 说明原因
   ↓
3. 提供选项（继续/暂停/调整）
   ↓
4. 等待用户决策
```

## 🎯 最佳实践

### 计划书优化

1. **使用表格** - 信息一目了然
2. **添加 emoji** - 增强可读性
3. **分点说明** - 避免大段文字
4. **明确确认** - 加粗确认提示

### 执行优化

1. **实时记录** - 不要事后回忆
2. **保存证据** - 截图/日志/输出
3. **及时沟通** - 问题不过夜
4. **备份中间结果** - 防止丢失

### 汇报优化

1. **数据说话** - 用表格展示统计
2. **诚实透明** - 不隐瞒问题
3. **提供建议** - 不只是汇报
4. **感谢用户** - 保持专业态度
