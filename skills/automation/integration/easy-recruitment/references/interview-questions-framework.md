# 面试问题生成框架

针对不同类型的简历内容，我会生成对应的问题清单。

---

## 问题类型A：技术能力验证

### 针对技术栈的问题

**示例场景**：简历写"熟悉Redis"
- ❌ 差问题：什么是Redis？
- ✅ 好问题：你在XX项目中用Redis解决了什么问题？为什么选择Redis而不是其他方案？
- 追问：Redis的数据持久化是怎么配置的？如果Redis挂了会怎样？
- 期望回答要点：
  - 具体业务场景（缓存/分布式锁/排行榜等）
  - 技术选型理由
  - 对潜在风险有考虑

**示例场景**：简历写"有高并发优化经验"
- 问题：能举一个你优化过的具体案例吗？优化前后的数据对比是什么？
- 追问：除了这个方案，你还考虑过其他方案吗？为什么没选？
- 期望回答要点：
  - STAR法则完整叙述
  - 量化的优化效果（QPS提升XX%，延迟降低XXms）
  - 对方案的权衡思考

### 针对项目经验的问题

**示例场景**：简历描述"负责订单系统的重构"
- 问题：这个重构的背景是什么？原来的系统有什么问题？
- 追问：重构过程中遇到的最大挑战是什么？怎么解决的？
- 追问：如果重来一次，你会怎么做 differently？
- 期望回答要点：
  - 对业务背景的理解
  - 技术决策的过程
  - 从项目中学到的经验

---

## 问题类型B：项目深挖（STAR法则）

### Situation（背景）
- 这个项目是解决什么问题的？
- 当时团队是什么情况？你是什么角色？
- 项目的难点/挑战是什么？

### Task（任务）
- 你的具体职责是什么？
- 项目的目标是什么？有哪些约束条件？
- 你负责模块的边界是什么？

### Action（行动）
- 你具体做了什么？（重点追问）
- 为什么选择这个方案？有没有考虑过其他方案？
- 过程中遇到了什么困难？怎么解决的？

### Result（结果）
- 最终结果如何？有数据吗？
- 得到了什么反馈？
- 你学到了什么？

## 问题类型C：软实力评估

### 基础素质评估（基于 personalprefer.txt）

如果面试官在 personalprefer.txt 中明确列出了基础素质要求，我会针对这些要求设计专门的问题：

#### 1. 聪明 / 学习能力强
- 你最近学到的一个最有用的知识/技能是什么？是怎么学会的？
- 如果让你一周内掌握一门新技术，你会怎么规划？
- 期望回答要点：有学习方法论、能快速抓住核心、有实际成果

#### 2. 逻辑思维清晰
- 请描述一个复杂问题，你是如何拆解并解决的？
- 如果让你向一个完全不懂的人解释你的项目，你会怎么说？
- 期望回答要点：结构化思维、条理清晰、能化繁为简

#### 3. 沟通表达能力强
- 描述一次你需要说服别人接受你观点的经历。
- 如果团队成员对你的方案有质疑，你会怎么处理？
- 期望回答要点：有说服力、善于倾听、能找到共识

#### 4. 踏实肯干 / 执行力强
- 描述一个你认为很枯燥但必须完成的任务，你是怎么完成的？
- 当项目进展不如预期时，你会怎么做？
- 期望回答要点：不挑活、能落地、有韧性

#### 5. 主动积极 / 有 ownership
- 有没有过你发现某个问题但没人指派，你主动去解决的经历？
- 如果项目出了错，但不是你的责任，你会怎么处理？
- 期望回答要点：主动担责、不推诿、有主人翁意识

#### 6. 抗压能力强
- 描述一次你承受很大压力的经历，你是怎么调节的？
- 如果多个任务同时 deadline，你会怎么安排？
- 期望回答要点：情绪稳定、有应对策略、不倒苦水

#### 7. 团队协作好
- 描述一次你和团队一起攻克难题的经历，你扮演了什么角色？
- 如果团队成员拖后腿，你会怎么处理？
- 期望回答要点：有团队意识、能配合、也能补位

#### 8. 创新思维 / 敢想敢做
- 你有没有过什么"疯狂"的想法？后来实现了吗？
- 如果让你完全重新设计这个产品，你会怎么做？
- 期望回答要点：有想象力、敢尝试、也能落地

#### 9. 细节控 / 严谨认真
- 你有没有因为注意到了一个细节而避免大问题的经历？
- 你如何保证你交付的工作质量？
- 期望回答要点：有案例、有方法、不马虎

#### 10. 数据敏感 / 理性决策
- 描述一次你用数据说服别人的经历。
- 如果没有数据支撑，你会怎么做决策？
- 期望回答要点：用数据说话、逻辑严谨、不拍脑袋

#### 11. 用户导向 / 同理心强
- 描述一次你站在用户角度思考问题的经历。
- 如果用户反馈和老板意见冲突，你会怎么处理？
- 期望回答要点：能换位思考、有用户意识、能平衡各方

### 经典软实力问题

**团队协作**
- 描述一次你和同事有严重分歧的经历，最后怎么解决的？
- 如果你和一个很难合作的同事必须一起完成项目，你会怎么做？
- 期望回答要点：有具体例子、展示沟通和妥协能力、结果导向

**学习能力**
- 你最近学了什么新技术？是怎么学的？
- 如果明天要你用一门不会的语言写项目，你会怎么做？
- 期望回答要点：有持续学习习惯、有方法论、有输出（博客/项目）

**抗压能力**
- 描述一次项目deadline很紧的经历，你是怎么处理的？
- 如果线上出了严重bug，你会怎么处理？
- 期望回答要点：冷静分析、优先级判断、总结复盘

**主动性**
- 有没有过你觉得某个流程/代码可以优化，主动推动改进的经历？
- 期望回答要点：有owner意识、有推动力、有结果

## 问题类型D：风险点澄清

针对简历中的疑虑设计问题：

### 频繁跳槽
- 我看到你过去3年换了3份工作，能说说原因吗？
- 期望回答要点：有合理原因（如公司业务调整）、不是逃避问题、对未来有规划

### 经验断层
- 你有一年没有工作经验，这段时间在做什么？
- 期望回答要点：有合理解释、保持学习、与岗位相关

### 技能描述模糊
- 简历写"熟悉微服务"，能具体说说你用过哪些微服务组件吗？
- 期望回答要点：有真实使用经验、能说出优缺点

### 项目描述太泛
- 这个项目具体做了什么功能？你的贡献是什么？
- 期望回答要点：能讲清楚细节、突出个人贡献

## 回答评分标准

### 优秀回答（4-5分）
- 使用STAR法则，结构清晰
- 有量化数据支撑
- 展示深度思考（考虑过多种方案）
- 有反思和总结
- 诚实（不会的不装会）

### 合格回答（2-3分）
- 基本完整，但缺少细节
- 有数据但不够量化
- 能说明做了什么，但说不清楚为什么
- 缺少反思

### 差回答（0-1分）
- 答非所问
- 过于笼统（"我负责开发"）
- 明显夸大或造假
- 答不出来但硬撑

## 面试问题清单模板

每个候选人的报告会包含以下格式的问题清单：

```
## 面试问题清单 - 张三

### 技术能力（15分钟）
1. 【Redis】你在XX项目中提到用Redis做缓存，当时缓存穿透是怎么处理的？
   - 期望要点：提到布隆过滤器或空值缓存，了解缓存雪崩/穿透/击穿的区别

2. 【高并发】你做过接口性能优化，能说说具体怎么做的吗？优化效果怎么样？
   - 期望要点：有量化数据（如QPS从1000提升到5000），说明优化手段（索引/缓存/异步等）

3. 【Go语言】你看你主要用Go，能说说Goroutine和Channel的使用场景吗？
   - 期望要点：有实际使用经验，能说出优缺点

### 项目深挖（15分钟）
4. 【XX系统重构】这个重构的背景是什么？原来的系统有什么问题？
   - 追问1：重构过程中最大的技术挑战是什么？
   - 追问2：如果重来一次，你会怎么做 differently？
   - 期望要点：对业务理解深刻，技术决策有思考，有复盘总结

### 软实力（10分钟）
5. 【团队协作】描述一次你和同事有技术分歧的经历，最后怎么解决的？
   - 期望要点：有具体例子，展示沟通和妥协能力

6. 【学习能力】你提到最近在学K8s，能说说学到什么程度了吗？有实践吗？
   - 期望要点：有学习计划，有输出（博客/项目）

### 风险点澄清（5分钟）
7. 【频繁跳槽】过去3年换了3份工作，能说说原因吗？
   - 期望要点：有合理原因，对未来有规划

---
预计总时长：45分钟
```

## 面试官偏好影响（personalprefer.txt）

如果 `jobs/` 根目录存在 `personalprefer.txt`，我会根据其中的配置调整问题设计。

**重要**：
- **personalprefer.txt** 位于 `jobs/` 根目录，全局共享
- 描述面试官的固定性格和偏好，不随岗位变化
- **JD.txt** 位于各职位文件夹内，描述岗位要求，随岗位变化

### 1. 面试官风格适配

**理性分析型面试官**：
- 更多数据、逻辑、结构化的追问
- 强调量化成果和决策过程
- 示例："你提到提升了效率，具体数据是什么？怎么测量的？"

**感性直觉型面试官**：
- 更多动机、价值观、团队契合度的问题
- 强调个人成长和职业选择
- 示例："为什么选择这个方向？是什么吸引你？"

**务实结果导向面试官**：
- 聚焦成果和交付
- 关注解决实际问题的能力
- 示例："这个项目最后的数据怎么样？如果再做一次会怎么优化？"

**开放探索型面试官**：
- 鼓励发散思维
- 接受非标准答案
- 示例："如果没有任何约束，你会怎么设计这个产品？"

**严格细节控面试官**：
- 深入技术细节
- 关注边界情况和异常处理
- 示例："这种情况下边界条件是怎么处理的？有没有考虑过失败场景？"

### 2. 偏好素质重点考察

根据 personalprefer.txt 中勾选的素质要求，我会：

1. **增加对应素质的考察权重**
   - 如果勾选了"沟通表达能力强"，会增加沟通场景的问题
   - 如果勾选了"聪明，学习能力强"，会增加学习场景的问题

2. **调整期望回答要点**
   - 对重要素质，期望回答中必须包含相关证据
   - 对其他素质，作为参考项而非必须项

3. **设计"必考题"**
   - 如果面试官在 personalprefer.txt 中列出了必考问题，会直接包含在问题清单中
   - 针对候选人背景定制这些必考题

### 3. 避免的提问方式

根据面试官不喜欢的候选人表现，我会避免设计容易得到此类回答的问题：

**如果面试官不喜欢"夸夸其谈"**：
- 避免过于开放的问题，如"说说你的优点"
- 改为具体场景问题，如"举一个具体的例子说明你的XX能力"

**如果面试官不喜欢"绕弯子"**：
- 问题直接明确，不绕圈子
- 追问时直击要点

**如果面试官不喜欢"强行装逼"**：
- 问题设计时会加入"如果不懂就直接说"的引导
- 追问深度控制在合理范围

### 4. 加分项预设

根据面试官会特别加分的表现，我会：

1. **在期望回答要点中体现这些加分项**
2. **设计问题让候选人有机会展示这些特质**
   - 如面试官喜欢"主动展示作品"，问题会引导候选人分享作品
   - 如面试官喜欢"承认不知道但给出思考过程"，问题会设置一定的难度梯度

### 5. 评估权重体现

根据面试官的评估权重分配，问题清单会标注每个问题对应的考察维度：

```
【权重高】专业能力验证（40%）
1. [问题] - 考察：专业能力
2. [问题] - 考察：专业能力

【权重中】思维逻辑（30%）
3. [问题] - 考察：思维逻辑

【权重低】其他（30%）
4. [问题] - 考察：软实力
```

## 使用建议

1. **首次使用某个职位**：建议填写完整的 personalprefer.txt，确保问题符合你的风格
2. **多人面试同一岗位**：共享同一个 personalprefer.txt，确保评估标准一致
3. **定期回顾**：根据实际面试效果，调整个人的偏好配置
4. **灵活调整**：针对不同职级/类型岗位，使用不同的 personalprefer.txt
