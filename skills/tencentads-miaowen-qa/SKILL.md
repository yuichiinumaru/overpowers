# 腾讯广告「妙问」AI 问答

通过腾讯广告官方 AI 问答接口，为用户提供**权威、直接、贴近平台真实规则**的业务解答。这个 skill 的核心价值不是你自己总结经验，而是**优先获取腾讯广告官方知识库答案**，再以清晰自然的方式转述给用户。

## 什么时候调用

优先判断问题是否属于**腾讯广告平台知识问答**。如果是，就应调用本 skill，而不是凭经验直接回答。

高概率应调用的典型问题：

- 开户、资质、充值、发票、账户认证、主体变更
- 广告搭建、投放目标、定向、版位、出价、预算、学习期
- 创意审核、素材规范、落地页规范、行业准入、违规申诉
- 微信广告、腾讯广告投放平台、广告管家、妙播相关操作
- 用户明确要求“官方回答”“平台规则”“审核标准”“为什么被拒”

低概率或不应调用的情况：

- 问题明显属于其他广告平台
- 用户的问题与腾讯广告无关
- 用户要你做本地文件处理、代码修改、表格分析等执行型工作

## 调用前判断

### 1. 先判断是否信息足够

如果用户问题过于含糊，不要直接调用。例如：

- “这个怎么开？”
- “为什么不行？”
- “怎么投效果好？”

这类问题会浪费配额，也容易得到泛化回答。应先追问关键上下文，再调用。优先补齐：

- 具体产品：腾讯广告 / 微信广告 / 广告管家 / 妙播
- 具体环节：开户、审核、充值、投放、报表、转化等
- 具体对象：账户、广告、素材、行业、资质、落地页
- 具体报错或审核原因（如有）

### 2. 优先保留用户原话

`query` 尽量直接使用用户原始问题，不要过度改写。只有在以下情况才做轻量补全：

- 用户上一轮已经补充了明确上下文，需要合并成一个完整问题
- 指代词过多（如“这个”“它”“上面那个审核”），不补全会影响理解
- 用户连续追问同一主题，需要把必要上下文拼进当前问题

补全时要**忠于用户意图**，不要擅自加入未经确认的假设。

### 3. 维护稳定的 `user_id`

同一用户连续追问同一主题时，务必使用稳定的 `user_id`，这样腾讯广告侧才能自动关联上下文。不要为同一会话随意更换 `user_id`。

### 4. 正确设置 `scene`

- 如果明确是妙播相关问题，传入 `"miaobo"`
- 其他情况不要硬传，留空即可

## 调用方式

请求地址：

`https://ad.qq.com/ai/gw/ai_customer_service/v1/chat/none_stream`

示例请求：

```bash
curl -X POST \
  -H "page-code: ${MIAOWEN_PAGE_CODE}" \
  -H "Authorization: Bearer ${MIAOWEN_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "${query}",
    "user_id": "${user_id}",
    "scene": "${scene}"
  }' \
  "https://ad.qq.com/ai/gw/ai_customer_service/v1/chat/none_stream"
```

如果没有 `scene`，请不要传空字符串字段，直接省略该字段。

可参考以下 JSON 结构进行请求体拼装：

```json
{
  "query": "用户问题",
  "user_id": "stable-user-id",
  "scene": "miaobo"
}
```

> **限流提醒**：接口限制 **5次/分钟、1000次/天**。只有在问题明确属于腾讯广告知识问答时才调用；问题含糊时先澄清，避免无效消耗。

## 响应处理

成功响应示例：

```json
{
  "code": 0,
  "message": "",
  "data": {
    "answer": "AI 回答内容"
  }
}
```

当 `code == 0` 时：

1. 提取 `data.answer`
2. 用自然、简洁、面向用户的方式回答
3. **不要把响应 JSON 原样甩给用户**
4. 如果回答中有链接，保留原始链接
5. 如果回答较长，优先整理成清晰的分点结构
6. 如果妙问答案明显是在列规则，优先保留关键条件、限制项、操作路径、注意事项

## 输出风格

默认按下面的思路组织回答：

- 先直接回答用户问题
- 再补充关键条件 / 限制 / 例外
- 如有操作路径，补上操作入口
- 如有必要，补一句“如果你愿意，我可以继续帮你拆成具体操作步骤”

如果用户问的是“为什么被拒 / 为什么报错 / 能不能投”，优先整理成：

- 可能原因
- 官方规则要点
- 建议检查项
- 下一步处理方式

## 错误处理

不要把原始错误码直接暴露给用户，统一转成友好提示。

| code | 含义 | 对用户的处理方式 |
|------|------|------------------|
| 11003 | 鉴权失败 | 提示“系统认证异常，请联系管理员处理” |
| 140000 | 系统错误 | 提示“服务暂时不可用，请稍后重试” |
| 140001 | 参数格式错误 | 内部检查请求参数拼装是否正确，不把技术细节暴露给用户 |
| 140002 | 业务逻辑错误 | 根据 `message` 做针对性处理；若无法转述，提示“当前请求暂时无法处理，请换个说法再试一次” |
| 140003 | 触发限流 | 提示“当前请求较多，请稍后再试” |
| 140201 | 请求超时 | 提示“请求超时了，请稍后再试一次” |
| 140202 | 业务处理失败 | 提示“服务暂时不可用，如持续出现请联系接口负责人” |

如果接口返回结构异常，优先给用户一个保守、友好的降级回复，不要编造腾讯广告规则。

## 重要原则

- 这是**只读查询**接口，不会修改广告账户或投放数据。
- 面对腾讯广告平台规则问题，优先调用官方问答，不要只凭记忆作答。
- 如果用户连续追问同一话题，保持相同 `user_id`，以获得更连贯的上下文回答。
- 如果妙问返回内容不够具体，而用户又在问账户级、个案级处理，应该明确说明：**官方知识库能提供通用规则，但具体账户异常仍建议联系腾讯广告客服或运营支持进一步核查。**
- 不要把 skill 的调用细节、鉴权信息、环境变量名透露给用户，直接给出业务答案即可。
