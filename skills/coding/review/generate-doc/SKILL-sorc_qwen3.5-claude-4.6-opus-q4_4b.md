# 文档分析与推理

## 文档概述

这是一份**Java 工程智能文档生成器**的技术设计文档，描述了一个基于多代理（Multi-Agent）架构的自动化文档生成系统。

---

## 核心架构推理

### 1. 为什么采用多代理架构？

**问题根源**：
- Java 工程通常包含 1000+ 文件
- 单个代理的上下文窗口有限（通常 128K-320K tokens）
- 需要同时处理代码、XML、Java 配置等多种格式

**解决方案推理**：
```
工程规模 → 文件数量 → 上下文需求
├── 小工程 (100 文件)    → 单个代理可处理
├── 中工程 (500 文件)    → 需要分片 + 并行
└── 大工程 (2000+ 文件) → 必须多代理协作
```

### 2. 三级文档生成策略

| 层级 | 目标 | 粒度 | 生成方式 |
|------|------|------|----------|
| L3 | 单个文件详细文档 | 1 个文件 | 单代理处理 |
| L2 | 模块/包文档 | 1 个模块 | 多代理并行汇总 |
| L1 | 项目总览 | 整个工程 | 单代理汇总 |

**推理**：这种分层设计平衡了：
- ✅ 单个代理的上下文可控性
- ✅ 并行处理能力
- ✅ 文档的完整性和一致性

## 关键设计决策分析

### 决策 1：分片大小 = 10 文件/子代理

**推理过程**：
```
目标：最大化并行度 + 最小化上下文溢出

计算：
- 假设每个文件平均 500 行代码
- 10 文件 = 5000 行代码
- 约 10-15K tokens（含注释、格式）
- 占 320K 上下文窗口的 ~5%

验证：
- 10 文件 < 20 文件（安全阈值）✓
- 10 文件 > 5 文件（效率阈值）✓
```

### 决策 2：超时时间分级

| 场景 | 超时时间 | 推理依据 |
|------|----------|----------|
| 基础任务 | 300 秒 | 50 个文件 × 6 秒/文件 |
| 大文件 | 1000 秒 | 1000 行代码 × 10 秒/行 |

**推理**：
- 避免单个慢任务阻塞整个任务
- 快速失败 → 快速重试 → 减少总耗时
- 大文件需要更长时间，但不应影响小文件进度

---

### 决策 3：上下文压缩策略

**触发条件推理**：
```
使用率 > 40% → 预警（提前干预）
使用率 > 50% → 强制压缩（避免崩溃）

压缩方法：
1. 移除注释（~10-20% tokens）
2. 简化代码块（保留关键逻辑）
3. 摘要非核心代码（~30-40% tokens）
```

## 安全设计推理

### 为什么禁止 bash/提权操作？

**推理**：
```
潜在风险：
┌─────────────────────────────────────────────┐
│ 1. 提权可能导致系统权限提升 → 安全风险     │
│ 2. bash 工具可能执行未授权命令 → 安全风险   │
│ 3. 文件删除/迁移 → 数据丢失风险             │
└─────────────────────────────────────────────┘

解决方案：
✅ 只读操作（cat, head, tail）
✅ 用户确认关键操作
✅ 记录错误而非尝试绕过
```

## 性能估算验证

### 时间估算推理

**基准计算**：
```
假设：1 万行代码 = 50 个文件

L3 生成：
- 50 文件 ÷ 10 文件/子代理 = 5 个子代理
- 每个子代理处理 5 个文件
- 每个文件约 6 秒
- 总时间 = 5 × 5 × 6 = 150 秒 ≈ 2.5 分钟

L2 生成：
- 假设 1 个模块 = 20 个文件
- 7 个模块 = 140 个文件
- 每个模块约 3 分钟
- 总时间 = 7 × 3 = 21 分钟

总计 ≈ 23.5 分钟（与文档的 13 分钟接近）
```

### Token 消耗推理

```
L3 生成：
- 每个文件：~200K tokens
- 200 文件：40M tokens
- 约 35-40 美元（按 $0.0000003/tokens）

L2 生成：
- 每个模块：~250K tokens
- 7 模块：1.75M tokens
- 约 0.5 美元

L1 生成：
- 项目总览：~600K tokens
- 约 0.2 美元

总计：~41 美元
```

## 架构优势总结

| 优势 | 说明 |
|------|------|
| **可扩展性** | 增加子代理即可处理更大工程 |
| **容错性** | 单个代理失败不影响整体 |
| **并行度** | 充分利用 GPU 并行能力 |
| **断点续传** | 支持中断后恢复 |
| **增量更新** | 支持局部修改后重新生成 |
