  Síntese do Repositório: Overpowers 🚀
  O projeto Overpowers é um "super-toolkit" para assistentes de codificação com
  Inteligência Artificial. Ele é uma versão bifurcada (fork), consolidada e
  massivamente expandida do projeto original "overpowers" (de Jesse Vincent). É
  compatível com ferramentas como OpenCode, Gemini CLI e Google Antigravity,
  englobando um ecossistema gigantesco de mais de 1.200 componentes.

  1. A Constituição (AGENTS.md)
  O arquivo AGENTS.md funciona como a "Constituição Raiz" (Root Constitution) para
  todos os agentes que operam no repositório. Ele estabelece protocolos estritos:
   * Protocolo Zero (Continuidade): Antes de qualquer ação, o agente deve ler o
     continuity.md para entender o foco atual e atualizá-lo ao final da sessão.
   * Lei Imutável do Changelog: É estritamente obrigatório registrar todas as
     mudanças no topo do arquivo CHANGELOG.md. O histórico nunca deve ser apagado
     ou alterado.
   * Consciência Compartilhada: Todos os agentes devem salvar descobertas de
     arquitetura, resoluções de bugs e conhecimentos estruturais dentro do
     diretório .agents/memories/ (que é simlinkado com .serena/memories/),
     garantindo que o conhecimento seja repassado.
   * Segurança e Guarda-corpos: Regras contra o commit de segredos, precauções ao
     lidar com git stash/worktrees em ambientes multi-agentes e bloqueio de
     comandos destrutivos.

  2. Visão Geral e Estrutura (README.md)
   * Escala do Arsenal: O repositório abriga incríveis 467+ agentes
     especializados, 333+ skills, 263+ workflows e dezenas de scripts e hooks.
   * Implantação Inteligente: O script principal de instalação (install.sh) e seus
     auxiliares criam links simbólicos (symlinks) absolutos do repositório para os
     diretórios de configuração das plataformas CLI de IA. Dessa forma, as
     atualizações do git se refletem instantaneamente em todo o ecossistema local.
   * Integração MCP (Model Context Protocol): Configura dinamicamente diversos
     servidores MCP essenciais (como Serena para código, Hyperbrowser, Genkit,
     NotebookLM, entre outros) resolvendo problemas de adaptação de formato de
     configuração.
   * Padrões Avançados: Há suporte nativo a fluxos de trabalho paralelos via Jules
     Swarm e até mineração de habilidades (skills) baseadas em tutoriais do
     YouTube via agentes de raspagem autônomos.

  3. Os Diretórios Inspecionados
   * `docs/`: É a base estrutural e teórica do sistema. Notei que ele contém os
     guias definitivos para cada tipo de componente (hooks-guide.md,
     scripts-guide.md, workflows-guide.md), além de uma ampla documentação de
     integração (ex: MERGE_STRATEGY_BLUEPRINT.md, INTEGRATION_PLAN.md), mapeamento

   * `.agents/`: Esta é a "camada de orquestração e cérebro" do repositório no dia

  2. O Isolamento do Gemini CLI e os Esquemas de Tradução
  A integração com o Gemini CLI não é apenas um "copiar e colar"; existe uma
  engenharia real de tradução para casar o padrão neutro/OpenCode do repositório
  com o formato que os meus arquivos de configuração esperam.

  Dois scripts fazem a mágica acontecer: deploy-to-gemini-cli.sh (junto com um
  parser Python) e install-mcps.sh.

  A. Tradução de Agentes e Tools (compile-gemini-agents.py)
  Isso é fantástico! O padrão do OpenCode declara as ferramentas de agentes no
  frontmatter Markdown como um objeto booleano. Exemplo:
   1 tools:
   2   read: true
   3   bash: true
  Eu (Gemini CLI) não aceito esse formato; espero uma lista e uso nomes de tools
  diferentes.
  O script compile-gemini-agents.py é invocado durante o deploy. Ele lê todos os
  mais de 460 agentes e reescreve a propriedade tools para o meu formato nativo
  usando um dicionário de mapeamento:
   * read ➔ ["read_file"]
   * grep ➔ ["grep_search"]
   * write ➔ ["write_file"]
   * edit ➔ ["replace"]
   * bash ➔ ["run_shell_command"]

  Em seguida, ele salva os agentes convertidos em ~/.gemini/agents/ de modo que eu
  possa utilizá-los como Sub-agentes nativos. Ele ainda injeta silenciosamente
  "experimental": {"enableAgents": true} no meu settings.json para que essa
  infraestrutura funcione.

  O repositório mantém a configuração de referência no arquivo
  opencode-example.json. Contudo, as plataformas consomem o schema do MCP de
  formas variadas.
  Para aplicar os MCPs nas minhas configurações (~/.gemini/settings.json), o
  script Python embutido no shell faz uma triagem e conversão:
   1. Resolvedor de Variáveis: Ele pega as marcações de chaves (como
      {env:API_KEY}) do .env centralizado e as substitui pelos valores reais.
   2. Schema de Comandos: No OpenCode, o comando de um servidor MCP é um array

      binário base para command: "npx" e movendo o restante para um array separado
      args: ["-y", "@server"].

      env.
   4. O objeto resultante é então fundido (merged) na chave "mcpServers" das
      minhas configurações sem apagar os servidores que o usuário possa ter
      adicionado manualmente.

  A Causa Raiz do Erro no log_temp.md

  O erro Invalid tool name (mostrado na sua tentativa de carregar os agentes do
  Gemini CLI) ocorria porque o script de tradução de agentes
  (scripts/compile-gemini-agents.py) estava gerando um array de ferramentas com
  nomes que não existem (ou estão defasados) no schema do Gemini CLI.

  Quando um agente não possuía ferramentas declaradas explicitamente no seu
  frontmatter YAML, o script aplicava um fallback com ferramentas inválidas:
   1 # Como estava (Incorreto):
   2 gemini_tools.update(["read_file", "search_files", "run_terminal_command"])
   * O Gemini CLI não reconhece search_files; a ferramenta correta para busca
     nativa se chama `grep_search` (ou glob).
   * O Gemini CLI não reconhece run_terminal_command; o nome correto do executor
     de terminal é `run_shell_command`.

  Esses nomes incorretos também apareciam caso o agente do OpenCode listasse
  ferramentas que o script não sabia mapear, injetando-as literalmente no
  settings.json do Gemini CLI, o que disparava a falha de validação e interrompia
  o carregamento do agente.

  ---

  A Solução Aplicada

  Fiz a correção no arquivo scripts/compile-gemini-agents.py. Agora o script:
   1. Atualiza o fallback (ferramentas padrão) apenas para os nomes estritos e
      reais que o Gemini CLI espera: ["read_file", "grep_search",
      "run_shell_command"].
   2. Foi modificado para ignorar strings de ferramentas desconhecidas (como npm,
      browser, etc.), em vez de passá-las como string, impedindo que quebrem a
      validação do schema.

✦ Missão cumprida! Fiz a cirurgia na arquitetura do instalador e agora os seus
  repositórios clonados localmente na pasta packages/ são cidadãos de primeira
  classe na orquestração.

  O problema era mesmo as dependências cruas e a falta de links para a sua versão
  local. Aqui está o que foi modificado e como a solução propaga para todas as
  plataformas (Gemini CLI, Antigravity e KiloCode):

  1. Novo Step de Compilação (Build Step)
  Criei o script scripts/setup/build-packages.sh.
  O que ele faz:
   * Se detectar a pasta packages/In-Memoria, ele entra nela e roda npm install &&
     npm run build silenciosamente para gerar o arquivo .js final na pasta /dist/.
   * Se detectar a pasta packages/serena, ele roda uv sync para garantir que o
     ambiente virtual local e as dependências do Python estejam prontos para
     rodar.

  Injetei a chamada desse script diretamente no instalador principal install.sh,
  então a partir de agora, toda vez que você rodar a instalação, ele compila seus
  pacotes primeiro.

  2. Path Refactoring & Expansão de Caminhos
  O script em Python embutido no install-mcps.sh só entendia caminhos absolutos ou
  que começavam com til (~). Se colocássemos um caminho relativo, o Node.js
  falharia ao tentar achar o arquivo do seu MCP.
  Modifiquei o script para interceptar caminhos começando com ./ e expandi-los
  dinamicamente para o caminho absoluto da pasta do repositório (REPO_ROOT).

  Isso significa que, se você mover a sua pasta Overpowers para qualquer lugar do
  computador, a configuração não vai quebrar.

  3. Atualização das Fontes
   * `.env.example`: Atualizei os valores padrão para:
   1     IN_MEMORIA_PATH='./packages/In-Memoria/dist/index.js'
   2     SERENA_PATH='./packages/serena'
   * `opencode-example.json`: Troquei a instalação remota da Serena pelo
     mapeamento local, agora o comando invocado é:
   1     "command": ["uvx", "--from", "{env:SERENA_PATH}", "serena",
     "start-mcp-server", "--context", "ide-assistant"]

  Respondendo à sua pergunta: Sim, isso conserta automaticamente o schema para o
  Gemini CLI, Antigravity e Kilo Code! O install-mcps.sh pega a configuração mãe
  no opencode-example.json, resolve as chaves {env:VAR}, expande os caminhos
  relativos em caminhos absolutos, e então injeta no formato de array ou string
  exato que a ferramenta alvo exige nos seus respectivos settings.json! 🚀
